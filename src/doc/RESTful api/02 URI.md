URI
=====
RESTful APIにおけるURI（Uniform Resource Identifier）は、ネットワーク上のリソースを一意に識別するための文字列です。これはURL（Uniform
Resource Locator）またはURN（Uniform Resource Name）のいずれか、またはその両方を含むことができます。

たとえば、RESTfulなウェブサービスの場合、各リソースは一意なURLを持ちます。次の例では、URLはユーザーリソースと特定のユーザーを識別します:

https://api.example.com/users（全ユーザーを表すリソース）

https://api.example.com/users/123（IDが123のユーザーを表すリソース）

これらのURL（ここでいうURI）はそれぞれのリソースを指し、HTTPメソッド（GET、POST、PUT、DELETEなど）と組み合わせることで、リソースへの特定の操作（取得、作成、更新、削除など）を表します。

したがって、RESTful
APIにおけるURIはそのAPIの基本的な設計と実装において重要な役割を果たします。リソースを適切に識別し、そのリソースに対する操作を定義することで、APIの使用者にとって理解しやすく、使いやすいAPIを実現することができます。

ERD（Entity Relationship
Diagram）の粒度は、プロジェクトや目的によります。大規模なシステムや複雑なビジネスロジックを持つ場合、各エンティティやリレーションシップがどのように動作し、どのように関連しているかを示すために詳細なERDが有用です。一方で、小規模なプロジェクトでは、そこまで詳細でないERDでも十分かもしれません。重要なのは、ERDがシステムの全体像を捉え、理解しやすいことです。

凝集度が高いモジュールとは、そのモジュール内の要素や機能が密接に関連し、連携して動作するものを指します。その関係性は共通の目的、機能、データ、あるいはそれらの一部によってまとめられるべきです。凝集度が高いモジュールは、再利用やメンテナンスが容易であり、意図したとおりに動作する確率が高いです。

分散トレーシングとは、マイクロサービスアーキテクチャや分散アーキテクチャにおけるシステムの振る舞いを理解し、監視するための手法です。各サービスが独立して動作しているシステムでは、一つのエンドユーザー要求がどのように各マイクロサービスを通り抜けるかを特定しスローダウン問題を解析することが可能となります。これを実現する一つの方法は、一つのリクエストがシステム全体を通過する際にそれを追跡する一意なIDを使用することです。
